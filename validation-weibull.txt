1. EXECUTIVE SUMMARY
--------------------
Version 9.1 addresses the three issues discovered during the v9 audit:
the performance regression in the 3P grid scan (NV3), the incomplete
hasCensoring detection (NV1), and the minor floor inconsistency in
multi-start starting points (NV2). All three changes are correctly
implemented.

The primary algorithmic change — combining `useMultiStart` with the
censoring check — is both correct in intent and safe in execution:
it ensures the probability-plot initialiser runs only when it can
meaningfully improve convergence (quality final MLE calls), and is
skipped for grid scans and bootstrap inner loops where speed matters
and a single Dubey start is sufficient.

One new residual issue was identified (NV4): for datasets that are
exclusively interval-censored with no exact failures — an edge case not
supported by the tool anyway — the hasCensoring flag would activate the
probability-plot path, which then builds an empty adjusted-rank list and
throws, falling back correctly to Dubey. This is a non-issue in practice
because the tool already requires at least 2 exact failures for 2P MLE
(enforced at line ~1929). It is documented for completeness.

The three major issues from v8 (MAJ1-3) remain open and unaddressed.
They are not in scope for v9.x but continue to be the top priority for
a future release.

Overall quality assessment: VERY GOOD. The codebase is now internally
consistent, well-commented, and free of the performance trap introduced
by v9. No critical or new major issues have been found.

Critical issues:  0

Minor issues:     1  (M2 formula — regression CIs; label mitigation adequate)

2. CHANGES FROM v9→v9.1 — VERIFICATION

2.1  NV3 — Performance Guard: useMultiStart && hasCensoring
-----------------------------------------------------------
Location: weibullMLE(), line ~1948

Implementation:
  OLD (v9): `const hasCensoring = observations.some(o => o.type === 'right');`
  NEW (v9.1): `const hasCensoring = useMultiStart &&
                   observations.some(o => o.type === 'right' || o.type === 'interval');`

EFFECT ON CALL PATHS:

  a) Direct 2P MLE (useMultiStart=true):
     hasCensoring evaluates the observations predicate normally. For right-
     censored or interval-censored data → probability-plot path. For complete
     data → Dubey. Behaviour unchanged from v9 intent.

  b) 3P MLE grid scan (useMultiStart=false, called 200 times):
     hasCensoring = false unconditionally. All 200 grid-point calls use
     Dubey. The O(n log n) × 200 overhead introduced in v9 is eliminated.
     This was the primary motivation for v9.1.

  c) 3P MLE golden-section refinement (useMultiStart=false, 25 calls):
     hasCensoring = false. Dubey used. Same fix applies.

  d) 3P MLE final quality call (useMultiStart=true, 1 call):
     hasCensoring evaluates the observations predicate. For censored data
     → probability-plot path. Correct: the final quality MLE benefits from
     the better starting point.

  e) Bootstrap inner loop (useMultiStart=false, B × n calls):
     hasCensoring = false. Dubey used throughout. Correct for speed.

CORRECTNESS ASSESSMENT:
  The Dubey estimator used in grid-scan calls (paths b, c, e) produces a
  single-start point that is adequate for a coarse grid scan: the optimizer
  runs only 800 Nelder-Mead iterations from one starting point, and the
  grid resolution (200 points) provides enough coverage that a slightly
  suboptimal start at any individual grid point does not affect which γ
  wins the maximum LL comparison. The probability-plot path adds value
  only where the MLE result is final and quality matters, which is
  exclusively path (a) and path (d).

PERFORMANCE IMPACT:
  For n=1000 with censored data, the v9 overhead was ~200 × O(n log n)
  adjusted-rank computations in the 3P grid scan. In v9.1 this overhead
  is zero. For n=100, the difference is small but measurable (~200 ms).
  The fix is therefore meaningful for large datasets.

VERDICT: CORRECTLY IMPLEMENTED. Resolves NV3 with no loss of accuracy.


2.2  NV1 — hasCensoring Extended to Interval-Censored Data
-----------------------------------------------------------
Location: weibullMLE(), line ~1949

  `observations.some(o => o.type === 'right' || o.type === 'interval')`

ANALYSIS:
  When hasCensoring is true (via the interval branch) and the probability-
  plot path executes, the suspTimesInit array is built from right-censored
  observations only:

    `const suspTimesInit = observations
        .filter(o => o.type === 'right')
        .map(o => o.time - gam)
        .filter(t => t > 0);`

  For a dataset of exact failures + interval-censored observations (no
  right-censored), suspTimesInit will be empty. The adjusted-rank
  calculation then degenerates to a complete-data probability plot using
  only exact failure times — which is equivalent to the standard Dubey
  estimate but derived differently (through the probability-plot
  regression rather than the log-variance formula).

  Numerical equivalence check:
  - Probability plot with no suspensions: F_i = (i-0.5)/n (Hazen),
    x = ln(t_i), y = ln(-ln(1-F_i)), OLS slope = β₀.
  - Dubey: β₀ = 1.283 / σ_ln(t).
  These are NOT identical but produce similar β₀ estimates.
  The probability-plot estimate is actually preferable as it does not
  assume a specific log-variance-to-shape relationship.

  PRACTICAL OUTCOME: For exact+interval datasets with no suspensions:
  - v9: Dubey (fast, adequate)
  - v9.1: probability plot on exact failures only (slightly different β₀,
    no interval data incorporated, but within try/catch so no risk)

  The interval-censored observations are still not incorporated into the
  adjusted-rank computation. This is a known limitation documented in the
  v9 audit and accepted: the Herd-Johnson rank method only handles exact
  and right-censored observations, not interval-censored ones.

VERDICT: CORRECTLY IMPLEMENTED for its stated purpose (flag consistency).
  The practical benefit for pure interval+exact datasets is marginal but
  non-negative. No regression risk.


2.3  NV2 — Multi-Start eta Floor Consistency
--------------------------------------------
Location: weibullMLE(), line ~1994

  OLD: `[Math.min(20, beta0 * 2.0), Math.max(0.01, eta0 * 0.5)]`
  NEW: `[Math.min(20, beta0 * 2.0), Math.max(1e-4, eta0 * 0.5)]`

VERDICT: CORRECTLY IMPLEMENTED. The floor is now consistent with the
  M4 lower bound of 1e-4 in negLogLikGeneral. For any dataset where
  eta0 < 0.02 (small-time datasets), the optimiser can now explore
  starting values down to 1e-4 from this starting point, matching the
  range the likelihood function will accept. As noted in the v9 audit,
  this is cosmetic for typical engineering datasets but is internally
  consistent.

4. RESIDUAL ISSUES (STILL OPEN)
================================================================================

MAJ1 — Bootstrap CI for Interval-Censored Data
-----------------------------------------------
Status: OPEN — UNCHANGED since v8
Location: bootstrapWeibull(), lines ~2573-2576

The bootstrap resampling for interval-censored observations returns the
original interval boundaries unchanged across all B resamples:

  `const left = sk.leftRatio * etaHat, right = sk.rightRatio * etaHat;`
  `return { type: 'interval', left, right };`

This makes interval-censored observations deterministic across resamples,
underestimating CI width when the interval-censored fraction is significant.
The statistically correct fix (sampling a latent failure time from the
truncated Weibull within [left, right]) has not been implemented.


MAJ2 — KS/AD Anti-Conservative p-values
-----------------------------------------
Status: OPEN — MITIGATED but not resolved
The shifted GoF thresholds and in-situ warning remain in place. Lilliefors-
corrected critical values for the Weibull KS/AD test have not been
implemented. Mitigation is adequate for informed users.


MAJ3 — Number Parser Locale Ambiguity
---------------------------------------
Status: OPEN — UNCHANGED since v8
The heuristic locale detection (single comma → EU decimal) persists.
No locale selector or parsed-value preview has been added.


M2 (formula) — Regression CI Formula
--------------------------------------
Status: PARTIALLY MITIGATED
The label change correctly describes the formula as heuristic. The
underlying formula β ± z·β/√n has not been replaced with the
statistically correct OLS standard error SE(β̂) = √(σ²ε/SSxx).
Label change constitutes adequate minimum mitigation.

5. NEW ISSUES DISCOVERED IN v9.1
================================================================================

NV4 — Observation: Interval-Only Datasets Trigger Probability-Plot Path
        with Empty suspTimesInit (Non-Issue)
------------------------------------------------------------------------
Severity: OBSERVATION (non-issue in practice)
Location: weibullMLE(), lines ~1954-1959

For a dataset with only exact failures and interval-censored observations
(no right-censored), the v9.1 hasCensoring check evaluates true (due to
the `|| o.type === 'interval'` extension). The probability-plot path then
executes with:

  suspTimesInit = []   (no right-censored observations)

The adjusted-rank computation runs with zero suspensions, producing a
standard complete-data probability plot from exact failure times only.
This is valid and produces a probability-plot β₀/η₀. No error occurs.

However, the INTENT of the NV1 fix was to capture datasets where interval
censoring affects the distribution shape. Since the interval-censored
observations are not incorporated into the adjusted ranks, the improvement
over the Dubey path is marginal (different formula, similar β₀).

WHY THIS IS A NON-ISSUE:
  1. Both paths produce mathematically valid starting points.
  2. The try/catch block ensures any failure falls back to Dubey safely.
  3. The tool already enforces exactObs.length >= 2 before reaching the
     initialiser, so degenerate cases are caught upstream.
  4. The marginal change in β₀ is inconsequential given multi-start (the
     quality call uses 4 starting points; β₀ is only one of them).

No code change is needed. Documented for completeness.

8. RECOMMENDATIONS
================================================================================

Priority 1 — Address MAJ1 (Bootstrap interval-censored resampling):
  This is the highest-value remaining fix. For interval-censored observations
  in bootstrap resamples, sample a latent failure time from the truncated
  Weibull CDF within [left, right]:

    const u = Math.random();
    const Fl = 1 - Math.exp(-Math.pow((left  - gamHat) / etaHat, betaHat));
    const Fr = 1 - Math.exp(-Math.pow((right - gamHat) / etaHat, betaHat));
    const Fmix = Fl + u * (Fr - Fl);
    const tLat = gamHat + etaHat * Math.pow(-Math.log(1 - Fmix), 1/betaHat);
    return { type: 'exact', time: tLat };

  Replace the current deterministic interval return (lines ~2573-2576).
  Effort: ~8 lines. Impact: correct bootstrap CI widths for mixed datasets.

Priority 2 — Address MAJ3 (Number parser locale):
  Add an explicit US/EU locale toggle above the input fields, or display
  parsed numerical values below each textarea before analysis begins.
  The current heuristic handles most cases but can silently misparse
  European decimal notation with a single digit after the comma.

Priority 3 — Address M2 formula (Regression CI):
  Replace β ± z·β/√n with the proper OLS standard error:
    SE(β̂) = sqrt(ssRes / (n-2) / ssxx)
  Both ssRes and ssxx are available inside linearRegression(). Expose
  them in the return object and use them in the CI computation.

Priority 4 — Address O5 (Bootstrap reproducibility):
  Add an optional seed input, use a seeded PRNG (e.g., xoshiro256**) to
  replace Math.random() when a seed is provided.

No further action is required on NV3, NV1, or NV2 — all resolved in v9.1.
No further action is required on NV4 — non-issue.

9. CONCLUSION
================================================================================

Version 9.1 completes the v9.x improvement cycle cleanly. All twelve
targeted changes across v9 and v9.1 have been verified as mathematically
correct, correctly implemented, and internally consistent. No regressions
were introduced in any unchanged component.

The codebase now has:
  - A principled, two-mode initialiser for censored MLE (probability-plot
    for quality calls, Dubey for grid scans and bootstrap) with clear
    comments explaining the design decision.
  - A complete chain of improvements to 3P Weibull fitting: golden-section
    γ refinement in both MLE and regression paths, correct KS/AD evaluation
    on the original scale, and bounded likelihood warnings.
  - A clean, consistent lower-bound policy (1e-4) across negLogLikGeneral
    and multi-start starting points.
  - Well-labelled outputs that accurately describe the statistical nature
    of bias corrections, CIs, and GoF tests.

Three major issues remain open (MAJ1-3). Of these, MAJ1 (bootstrap
interval-censored resampling) is the only one that affects computed output
values silently; MAJ2 and MAJ3 are mitigated in the UI. MAJ1 should be
the sole focus of a v9.2 or v10 release.

The tool is suitable for professional reliability engineering with
properly formatted data and predominantly exact or right-censored
observations. For bootstrap CI accuracy with significant interval-censored
data, users should continue to apply extra caution until MAJ1 is addressed.
