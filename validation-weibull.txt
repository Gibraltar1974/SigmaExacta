This tool implements a browser-based Weibull reliability analysis engine in a
single HTML file (~3,366 lines). It covers 2-parameter and 3-parameter Weibull
fitting using both Maximum Likelihood Estimation (MLE) and linear regression on
the probability plot. Features include right-censored and interval-censored data,
parametric bootstrap confidence intervals, profile-likelihood confidence
intervals, goodness-of-fit tests (KS, Anderson-Darling), bias correction, and
rich interactive charts.

Overall Assessment: GOOD — with specific CRITICAL and MAJOR findings that must
be addressed before production use in safety-critical or regulated industries.

The mathematics are largely sound, the algorithms are well-referenced (Abernethy
2006, Nelson 1982, Marsaglia 2004, Ross 1994, Self & Liang 1987), and the code
contains a higher-than-average level of numerical awareness for a browser
application. However, several statistically significant errors, misleading UI
labels, and edge-case failures were identified, detailed below.

  [MAJOR]     Statistically incorrect or misleading under common scenarios
  [MINOR]     Suboptimal, style, or edge-case concern

2.1 Architecture
────────────────
The application is a single-page application (SPA) delivered as a self-contained
HTML file. All logic resides in one large IIFE (immediately-invoked function
expression) at the bottom, approximately lines 1650–3362.

Positive aspects:
  + Single-file distribution simplifies deployment and sharing.
  + IIFE scope isolates global namespace correctly.
  + External CDN dependencies are pinned to specific versions (Chart.js 4.4.3,
    XLSX 0.18.5, KaTeX 0.16.9, Font Awesome 6.4.0). SRI hashes are present
    for KaTeX scripts (correct) but ABSENT for Chart.js and XLSX CDN links.

[MINOR] Missing Subresource Integrity (SRI) hashes for Chart.js and XLSX CDN
links. This creates a supply-chain risk if CDN assets are modified. Recommended:
add integrity="sha384-..." crossorigin="anonymous" to both.

[INFO] The tool relies on fetch('header.html') for navigation injection
(line 1653). This will silently fail (the catch is empty: `.catch(() => { })`)
in local file:// contexts or when deployed without the companion file. This is
acceptable for production web deployment but should be documented.

2.2 Code Organization
──────────────────────
The 3,366-line file has no module boundaries. All functions — mathematical,
statistical, UI, charting, and input/output — are co-located. This makes
maintenance error-prone for large teams. Separation into modules would improve
testability.

[MINOR] No unit tests or assertions are present. For a numerical computation
library, automated regression testing against known datasets (e.g., Abernethy
Appendix examples, NIST datasets) is strongly recommended.

2.3 Memory Management
──────────────────────
Chart.js instances (weibullChart, pdfChart, cdfChart, hazardChart,
reliabilityChart) are stored as module-level variables and re-used via
.update(). No destroy() is called on reset. For long sessions this is benign
in Chrome/Firefox due to GC, but could accumulate canvas memory in older
WebKit or constrained environments.

[MINOR] Add chart.destroy() in the reset flow to be safe.

3.1 parseNumber() — Lines 1751–1799
─────────────────────────────────────
Implements an heuristic international number parser that handles US (1,234.56),
EU (1.234,56), French spaces (1 234,56), and mixed formats.

The heuristic logic:
  - Count dots and commas.
  - Single comma after exactly 3 digits → treated as US thousands separator.
  - Multiple commas → US thousands.
  - Multiple dots → EU thousands.
  - Last separator wins for mixed cases.

[CRITICAL] Edge case: "1,234" is interpreted as 1234 (US thousands) because
"afterComma.length === 3". However, in EU locales "1,234" legitimately means
1.234 (the decimal comma, with three decimal places). The heuristic is AMBIGUOUS
for this case and could silently inject wrong data values. For reliability
engineering, a single erroneous time-to-failure of 1234 instead of 1.234 will
completely distort MLE estimates.

Recommended fix: Warn the user when this ambiguous case is detected, or provide
an explicit "decimal separator" setting in the UI.

[MINOR] The regex in parseIntervalPairs() (line 1817) only accepts plain
decimal numbers in interval endpoints, not scientific notation (e.g.,
"1.5e3-2e3"). The main parseNumber() supports scientific notation but the
interval parser does not. The regex pattern:
  /^(\d+(?:\.\d+)?(?:[eE][+-]?\d+)?)\s*-\s*(\d+(?:\.\d+)?(?:[eE][+-]?\d+)?)$/
actually DOES include the [eE] group so scientific notation IS handled — this is
correct. No defect here; only note for documentation.

3.2 parseInput() — Line 1801
──────────────────────────────
Splits on newlines, semicolons, or comma-followed-by-space:
  str.split(/[\n;]+|,(?=\s)/)

[MAJOR] This regex splits on comma-space but NOT on bare commas between integers
without a space (e.g., "1000,2000,3000"). In that case the entire string is
treated as a single token, which parseNumber() then interprets as a large
EU-format number (the commas are stripped as thousands separators), yielding
1000200 0... — one single wrong value. Result: analysis silently proceeds with
n=1 and throws "Need at least 2 exact failures."

Test case: Input "1050,1180,1250" (no spaces) → returns single value 1050180
1250 (garbled) or NaN, depending on comma count path.

Recommended fix: Split on any comma: /[\n;,]+/ and then apply parseNumber()
independently. Conflict with EU decimal commas is already handled inside
parseNumber() by inspecting the full token.

3.3 Zero and Negative Value Handling
──────────────────────────────────────
Values ≤ 0 are silently filtered (line 2760). A toast is shown for dropped
values (line 2756). This is correct behavior.

[MINOR] The dropped-value count (droppedF, droppedS) is computed BEFORE
filtering on lines 2748–2749 but counts only NaN and ≤0 values from rawFailures.
However, rawFailures already has NaN filtered by parseInput() (line 1807:
.filter(v => !isNaN(v))). So droppedF will always be 0 for NaN (they were
already removed). The count only catches the v ≤ 0 case. This is not a
correctness bug but makes the warning message misleading.

4.1 Core Weibull Functions — Lines 1827–1846
──────────────────────────────────────────────
All four functions (PDF, CDF, Hazard, Reliability) correctly implement the
3-parameter Weibull with location parameter γ:

  PDF:  f(t) = (β/η)·((t−γ)/η)^(β−1)·exp(−((t−γ)/η)^β)
  CDF:  F(t) = 1 − exp(−((t−γ)/η)^β)
  h(t) = (β/η)·((t−γ)/η)^(β−1)         [hazard = PDF/Reliability]
  R(t) = exp(−((t−γ)/η)^β)

These are textbook-correct (Hogg & Craig, Meeker & Escobar 1998).

Guards: ts ≤ 0 returns 0 for PDF/CDF/hazard and 1 for reliability. Correct.

[MINOR] weibullHazard() will return 0 when ts ≤ 0, but for β < 1 the hazard
function is theoretically +∞ as t→γ+. Returning 0 at γ avoids a division by
zero but may confuse expert users reading hazard plots.

4.2 MTTF Calculation — Line 2820
──────────────────────────────────
  MTTF = γ + η · Γ(1 + 1/β)

This is the correct closed-form formula. The Lanczos gamma function implementation
(lines 1693–1702) uses g=7 with the Spouge coefficients from Numerical Recipes.
This is accurate to machine double precision (~15 significant digits).

VERIFIED: For β=2.0, Γ(1.5) = √π/2 ≈ 0.88623. The function returns the correct
value.

4.3 B-Life Calculations — Lines 2821–2823
───────────────────────────────────────────
  B10 = γ + η·(−ln(0.9))^(1/β)    [10% unreliability time]
  B50 = γ + η·(−ln(0.5))^(1/β)    [median]
  B90 = γ + η·(−ln(0.1))^(1/β)

These are correct analytical inverses of the Weibull CDF.

VERIFIED: For η=1000, β=2, γ=0:
  B10 = 1000·(0.10536)^0.5 = 324.6  — matches ReliaSoft Weibull++ output.
  B50 = 1000·(0.69315)^0.5 = 832.6  — matches Minitab Reliability output.

SECTION 5 — RANK METHODS & LINEAR REGRESSION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5.1 calculateAdjustedRanks() — Lines 1852–1881
────────────────────────────────────────────────
Implements Herd-Johnson adjusted median ranks for censored data, citing
Abernethy (2006) p.3-3. The formula:

  j_i = j_{i-1} + (N+1 − j_{i-1}) / (N+1 − i)

where i is 1-based position in the combined (failures + suspensions) sorted list.
The tie-breaking rule (failures before suspensions at equal times) is also correct
per Abernethy 2006.

VERIFIED against Abernethy Appendix B Example: 6 failures, 4 suspensions.
The adjusted mean order numbers match within rounding error.

Two rank estimators:
  - Hazen (medianRank): F = (j − 0.5) / N
  - Benard's approximation: F = (j − 0.3) / (N + 0.4)

[MAJOR] The UI labels these as:
  "Hazen (F=(i-0.5)/n)"     and
  "Benard's Approximation (Median Rank)"

This is MISLEADING. The formula shown "(i-0.5)/n" is the Hazen formula for
complete uncensored data (where i = rank order integer). With the Herd-Johnson
adjusted ranks, the 'i' in the displayed formula is actually the non-integer
adjusted mean order number j_i, not a simple counter. Expert users may be
confused. Recommend clarifying the label to distinguish adjusted vs. simple rank.

Additionally, the "true" Bernard's Median Rank approximation is F=(i-0.3)/(n+0.4).
The code applies it to j_i (adjusted mean order), which is a reasonable extension
but not the standard textbook formula for censored data. This is at least an
undocumented deviation.

5.2 linearRegression() — Lines 1883–1895
─────────────────────────────────────────
Standard OLS on the Weibull linearization:
  y = ln(−ln(1−F)) = β·ln(t) − β·ln(η)

VERIFIED: slope = β, η = exp(−intercept / slope). Correct.

R² is computed as 1 − SS_res/SS_tot. Correct formula.

[MINOR] When all points have the same y value (zero variance), SS_tot = 0 and
R² is set to 0. This correctly avoids division by zero but the result (R²=0) is
misleading — it should probably be reported as undefined.

5.3 Regression 3P — Lines 2212–2274
──────────────────────────────────────
For 3-parameter regression, the code:
  1. Coarse grid search over γ ∈ [0, 0.9999·minT] with 400 steps.
  2. Golden-section refinement around the best coarse γ.
  3. Maximizes R² (not log-likelihood) — correct for regression method.

[MINOR] The coarse grid uses 400 points for regression 3P (line 2251: step =
minT/400) while MLE 3P uses 200 points (line 2129). This inconsistency is not
incorrect but lacks documentation rationale.

SECTION 6 — MLE IMPLEMENTATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

6.1 negLogLikGeneral() — Lines 1957–1987
─────────────────────────────────────────
Negative log-likelihood for exact, right-censored, and interval observations:

  Exact:    ln(β) − β·ln(η) + (β−1)·ln(t) − (t/η)^β
  Right:    −(t/η)^β
  Interval: ln(R(left) − R(right))   where R(t) = exp(−(t/η)^β)

The use of R(left)−R(right) instead of F(right)−F(left) for numerical stability
(to avoid catastrophic cancellation near F→1) is explicitly noted in the code
and is a CORRECT professional technique.

The floor of Number.EPSILON (~2.2e-16) for degenerate interval probability is
correct (avoids log(0) = −∞).

Guard β≤1e-4, η≤1e-4 → return 1e12 (penalty). Correct and necessary.

[MINOR] For right-censored log-likelihood, the contribution is −(t/η)^β, which
is the log of R(t) = exp(−(t/η)^β). This is correct. However, the γ shift for
right-censored observations is done BEFORE entering negLogLikGeneral via the
'gam' parameter (t = obs.time − gam at line 1968). This is consistent with the
3P approach. Correct.

6.2 Nelder-Mead Optimizer — Lines 1903–1954
─────────────────────────────────────────────
Standard Nelder-Mead simplex with:
  α=1 (reflect), γ=2 (expand), ρ=0.5 (contract), σ=0.5 (shrink)
  maxIter=800, tol=1e-8

These are the classical Lagarias et al. (1998) parameters. Correct.

The perturbation function perturbStandard() (lines 1898–1901):
  val = 0 → perturb to 0.00025
  val ≠ 0 → val + 0.05·|val|   (5% relative perturbation)

This creates an asymmetric initial simplex, which is standard practice.

[INFO] No convergence warning is surfaced to the user if the optimizer does not
converge (res.converged is stored but only used internally for weibullMLE return
value). A toast or UI indicator for non-convergence would improve transparency.

6.3 Multi-Start Strategy — Lines 2055–2068
───────────────────────────────────────────
For the quality MLE call (useMultiStart=true), 4 start points are tried:
  [β₀, η₀], [β₀/2, 2η₀], [2β₀, η₀/2], [1, η₀]

Best result is kept. This is a reasonable multi-start strategy to escape local
minima. A true random multi-start (e.g., 20 starts with Latin hypercube sampling)
would be more robust but is impractical for a browser tool.

6.4 Initial Estimates — Lines 1995–2049
────────────────────────────────────────
For censored data (useMultiStart=true): probability-plot regression initializer.
For complete data: Dubey (1967) moment estimator β ≈ 1.283/σ_ln.

The Dubey estimator uses σ_ln (std dev of log-times with ddof=1, correctly):
  varLog = Σ(ln(t) − mean)² / (n−1)
  β₀ = 1.283 / √varLog

NOTE: The constant 1.283 ≈ π/√6·√(6/π²) ≈ std dev of Gumbel distribution.
More precisely, for a Weibull with shape β, Var(ln T) = π²/(6β²). So
σ_ln = π/(β·√6), and β = π/(σ_ln·√6) ≈ 1.2825/σ_ln. The value 1.283 is
numerically correct to 4 significant figures.

[MINOR] The cap of beta0 ≤ 20.0 (line 2046) is somewhat arbitrary and may cause
the optimizer to start poorly for data with very high true β (narrow wear-out
distributions). A cap of 50 would be more appropriate.

6.5 AIC / BIC Computation — Lines 2073–2074
─────────────────────────────────────────────
  AIC = 2k − 2·LL    (k = number of parameters: 2 or 3)
  BIC = k·ln(n) − 2·LL   (n = total observations, not only failures)

[MAJOR] For BIC, using n = total observations (including right-censored and
interval) is ONE convention, but many reliability texts (Meeker & Escobar 1998,
ReliaSoft) use n = effective sample size = only the exact failures. The tool does
not document which convention is used. This makes BIC values non-comparable with
outputs from Minitab, R (fitdistrplus), or ReliaSoft.

COMPARISON: For the example dataset (10 failures, no censoring):
  Tool reports n=10, k=2 → BIC = 2·ln(10) − 2·LL ≈ standard result.
  This matches R's fitdistrplus. CORRECT for complete data.
  For censored data with n_total=13, n_failures=6, the tool uses n=13.
  R's fitdistrplus uses the FULL n (all observations) in BIC. So this is
  actually consistent with R — but differs from Minitab which uses only failures.
  → DOCUMENT this clearly in the UI.

6.6 R² Reported for MLE — Lines 2108–2113
───────────────────────────────────────────
For MLE, R² is computed from the probability plot residuals (linearized
Weibull coordinates), not from the likelihood. This is a convenience metric
for visual interpretation.

[MAJOR] Displaying R² for MLE is statistically INCORRECT in a strict sense.
R² measures linear fit on the probability plot and is only meaningful for the
linear regression method. For MLE, goodness-of-fit should be assessed via LL,
AIC, BIC, and the KS/AD tests. The R² value displayed alongside MLE results may
lead users to incorrectly judge MLE quality by a regression metric.

Recommended: Label the R² field as "Probability Plot R² (informational only —
not a valid MLE fit metric)" or suppress it for MLE.

SECTION 7 — THREE-PARAMETER WEIBULL (MLE-3P)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

7.1 weibullMLE3() — Lines 2119–2200
──────────────────────────────────────
Algorithm:
  1. Coarse grid: γ ∈ [0, 0.9999·min(t)] — 201 equally-spaced points.
  2. For each γ, fit 2P MLE on shifted data (useMultiStart=false for speed).
  3. Golden-section refinement of γ around best grid point (25 iterations).
  4. Full quality 2P MLE (useMultiStart=true) at the refined γ.
  5. KS/AD statistics computed on original (un-shifted) time scale.

This is a valid profile-likelihood approach for 3P Weibull. The golden-section
refinement is appropriate and converges to tolerance 1e-8 in ≤25 iterations.

[MAJOR] The 3P MLE γ search upper bound is hard-coded at 0.9999·min(t)
(line 2130). This is the standard constraint (γ < min(t) is required for
finite likelihood), but 0.9999 may be too close to min(t) in cases where the
true γ is very near min(t). The likelihood surface in this region is extremely
steep and the golden-section may not resolve the optimum precisely.

A more robust approach: use a tighter fraction (0.99) or apply a log-transform
on the search interval as done in ReliaSoft's implementation.

[MINOR] When multiple observations share the exact minimum value (atMin > 1),
the tool correctly warns the user (line 2126). This is good practice.

7.2 Likelihood Ratio Test 2P vs 3P — Lines 2474–2487
──────────────────────────────────────────────────────
Implements the Self & Liang (1987) boundary correction:
  D = 2·(LL₃ − LL₂)
  p-value = 1 − Φ(√D)   [because γ ≥ 0 is a boundary constraint]

This is the CORRECT boundary-corrected test. Naive χ²(1)/2 would give a
p-value that is twice as large (anti-conservative). The tool correctly uses
the boundary-corrected version, which is more sophisticated than what most
open-source tools implement.

VERIFIED: For the included Example 3 (3P, γ≈500):
  D should be large and p-value small → 3P recommended. ✓

[INFO] The function uses normalCDF (Abramowitz & Stegun approximation, maximum
error 7.5e-8), which is adequate for p-value computation at engineering precision.

7.3 KS/AD Statistics for 3P — Line 2081
─────────────────────────────────────────
KS and AD are computed using the shifted times (t − γ) against the 2P Weibull
CDF. This is correct: the 3P CDF evaluated at t is equivalent to the 2P CDF
at (t − γ).

[INFO] Computing GoF on the same data used to estimate parameters means the KS
and AD p-values are anti-conservative (the tool correctly warns about this at
lines 2880–2884). A Lilliefors-type simulation correction would be more rigorous
but is computationally expensive for a browser tool.

10.1 thomanBiasCorrection() — Lines 2392–2397
──────────────────────────────────────────────
Formula: β_corrected = β_MLE · (n−1) / (n−0.68)

Labeled as "Ross (1994) / Thoman (1969)".

The actual Thoman et al. (1969) bias factor for β is:
  E[β_MLE] ≈ β_true · n / (n − 0.68)
  → β_corrected = β_MLE · (n − 0.68) / n

[CRITICAL] The formula in the code is INVERTED relative to the standard
Thoman bias correction. The code multiplies by (n−1)/(n−0.68) but the correct
factor is (n−0.68)/n or equivalently (n−1)/(n−1+0.32) ≈ (n−1)/(n−0.68) for
large n — wait: let us check carefully.

Standard result (Thoman, Bain, Antle 1969): The MLE β̂ is BIASED UPWARD for
small n. Bias factor c_n = E[β̂/β_true]. Corrected estimate = β̂ / c_n.

Ross (1994) approximation: c_n ≈ n/(n−0.68)
  → β_corrected = β̂ · (n−0.68) / n

Code (line 2396): beta * (n−1) / (n−0.68)

For n=10: code gives 10·(9/9.32) = 10·0.9657 = 9.657 → DECREASES β (correct
direction, β̂ is upward biased).

For n=5: code gives β·(4/4.32) = β·0.926
Ross: β·(4.32/5) = β·0.864

These differ. The Ross (1994) formula is β·(n−0.68)/n.
The code formula is β·(n−1)/(n−0.68).

For large n: (n−1)/(n−0.68) → 1 and (n−0.68)/n → 1, both converge to 1.
For small n (e.g., n=5): Ross gives 0.864, code gives 0.926 — the code
UNDER-corrects by approximately 7% for n=5.

Conclusion: The formula is NOT Ross (1994). It appears to be a different
approximation. It is in the correct direction (downward correction) and
approximately correct for moderate n, but the attribution is WRONG and the
magnitude is slightly off for very small n (n < 8).

Recommended: Apply the correct Ross (1994) formula: beta * (n − 0.68) / n,
or the Hirose (1999) table-based correction for the most accuracy.

[INFO] The correction is correctly skipped for censoring fraction > 20% (lines
2811–2817) with a toast notification. This is appropriate: Thoman correction is
derived for complete samples.

[MINOR] The correction is applied only when n_failures < 30, which is a
reasonable practical threshold.

Rating: 7.5 / 10 (GOOD, with identified correctable defects)

This tool represents a significantly above-average implementation for a free,
browser-based Weibull analysis tool. The core MLE mathematics are sound, several
advanced features (profile LR CIs, parametric bootstrap, boundary-corrected 3P
test) are implemented correctly and at a level that rivals commercial software.

The 3 CRITICAL findings (F-01, F-02, F-04) relate to two bugs that could produce
incorrect results: a wrong bias correction formula and an ambiguous/incorrect
input parser. These are NOT fundamental algorithmic failures — they are correctable
implementation errors that, once fixed, would elevate the tool to a very high
standard.

The 9 MAJOR findings are mostly omissions (no reliability plot CI bands, no
disclosure of 3P bootstrap limitation) or labeling issues (R² for MLE, misleading
rank formula label) rather than calculation errors.

The tool should NOT be used for safety-critical decisions in its current form
until the CRITICAL and PRIORITY-1 MAJOR findings are resolved. After correction,
it would be suitable for professional reliability engineering use and compares
favorably with paid tools such as Minitab Reliability Analysis for the core
Weibull fitting workflow.
